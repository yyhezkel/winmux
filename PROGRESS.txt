TASK: Yossi tested the light directions and hit light-on-light /
  dark-on-dark surfaces. Root causes found and fixed — all three were
  the same class: values calibrated for dark-only that light grounds expose.
FIXES:
  1. PHANTOM CSS VARS (the big one): App.css referenced 11 var() names
     that were NEVER DEFINED — --w-text-primary, --w-text-secondary,
     --w-background, --w-bg-elev, --w-fg, --w-fg-muted, --w-mono,
     --w-surface-2, --w-surface-hover, --w-surface-raised, --w-warn —
     131 occurrences, every one silently falling back to a
     hardcoded-dark default. Invisible on dark themes (fallback ≈
     palette), broken on light. All renamed to the real tokens
     (--w-text / --w-text-dim / --w-bg / --w-surface / --w-surface-hi /
     --w-font-mono / --w-warning). +1 in ClaudeUsageIndicator.tsx.
     This fixes Yossi's two chrome reports: Add-ons "winmux-cli" title
     (light-on-light) and Settings→Logs viewer (dark-on-dark).
