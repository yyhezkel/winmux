# Decisions & Open Threads

Append-only log of ideas raised, decisions made, and threads in flight. Move entries between sections; don't delete history.

## How to use

Three sections:
- **Open** — questions or ideas awaiting a decision or action
- **Decided** — resolved entries with outcome
- **Archived** — things that became irrelevant or superseded

Entry format:

```
### YYYY-MM-DD — Short title
- **Context:** what came up
- **Options:** A / B / C (brief tradeoffs)
- **Status / Decision:** where things stand
- **Outcome / Commit:** hash + phase if implemented
```

When starting a session, scan **Open** first. Surface anything that's been pending too long.

### 2026-07-01 — Phase 72.3 — Monitor "daemon unreachable" root cause = `/current` blocked (daemon 1.2.4)
Yossi's debug.log gave the smoking gun: `insights_fetch: path=/logs http=200` **but** `path=/current http=000` — same daemon, same port/token, repeated. `http=000` under `curl --max-time 6` means the request never completed → the endpoint took **>6s**, not auth/liveness (`/logs` answered instantly). **Root cause:** `handleCurrent` ran `sm.Sample(true)` **synchronously in the request goroutine**, and a live sample can block — sequential per-container `dockerStats` (4s timeout × running containers), a `topProcesses` walk, or a hung `disk.Usage` mount — past the desktop's HTTP deadline. The Monitor read that timeout as "daemon unreachable" even though the daemon was healthy.
- **Fix (server-side, daemon 1.2.3→1.2.4):** (1) `Sampler` now caches the freshest snapshot (`last atomic.Pointer[Snapshot]`); the background ticker publishes it (ticker switched to `Sample(true)` so the cache carries top-processes). `handleCurrent` serves `sm.Current()` → **never blocks** on live collection; a slow/hung collector only staleness the cache, the endpoint still returns instantly. (2) `dockerList` fetches per-container stats **concurrently** (worst case = one stat's 4s timeout, was N×4s). (3) Timing self-diagnostic: a sample >2s logs `sample slow: total=.. cpu+mem+disk=.. docker=.. top=..` to insights.log (served by `/logs`) so the culprit phase is visible. Metadata only (Rule #1 safe).

### 2026-06-30 — v0.3.1 SHIPPED — critical pipe-instance-leak hotfix

### 2026-06-26 — (JJ) port-watcher leak — FIXED

### 2026-06-22 — Phase 65.O (REGRESSION): tmux scrollback no longer works
- **Finding (Yossi, v0.2.8 live test):** "מבחינת ה-TMUX - נראה כאילו הגלילה חזרה לא לעבוד - העתקה והדבקה כן עובדים".   2. **Shift+PgUp / Shift+PgDn** — should reach tmux to page copy-mode. A global `handleKey` (the Ctrl+B sidebar-cycle / `keyEq` additions from P62) may be capturing PgUp/PgDn even when a terminal is focused.

- **ROOT CAUSE CONFIRMED (2026-06-22, code read):** The custom key handler in `terminalInstance.ts` (`attachCustomKeyEventHandler`) only swallows Ctrl+C-for-copy — it does NOT translate keys. The arrow-key translation is **xterm.js's built-in alternate-scroll mode** (DECSET 1007): when the alternate screen buffer is active, xterm.js converts wheel ticks into cursor Up/Down. **tmux holds the alternate screen the entire time it runs**, so from xterm.js's view `alternate_on` is true even at a bare shell prompt inside tmux → wheel → Up/Down → shell history. The bundled `winmux-tmux.conf` comment (lines 21-23) *assumes* "wheel at a shell prompt = xterm.js local scrollback", which is **false inside tmux** — there's no xterm scrollback while tmux owns the alt-screen.
- **Why it's a regression:** Phase 60 (smoke-test #8) set `set -g mouse off` to restore native drag-selection/copy. With `mouse off`, the conf's `WheelUpPane`/`WheelDownPane` bindings (lines 42-43) are **inert** (wheel never reaches tmux), so nothing intercepts the wheel before xterm.js alt-scrolls it to arrows. Pre-Phase-60 (`mouse on`) the wheel reached tmux → copy-mode scroll = real scrollback.
- **The trade-off (Yossi's call):** this is fundamentally mouse-on (wheel scrollback works, but native left-drag selection needs Shift+drag to bypass tmux capture) **vs** mouse-off (native selection like today, but no wheel scrollback — need keyboard copy-mode bindings). The Phase 60 fix and this regression are two ends of the same lever. Options:
  - **(O-1) Keep `mouse off`; add keyboard scrollback.** Bind `S-PageUp`/`S-PageDown` (and optionally `S-Up`/`S-Down`) in the conf to enter/scroll tmux copy-mode (these are KEYS, so they reach tmux even with mouse off). Matches the DECISIONS "Shift+PgUp/PgDn → always scrollback" goal. Does NOT fix the wheel (wheel still alt-scrolls to arrows unless we intercept it in the xterm layer). Lowest risk to the copy/paste UX Yossi just got working.
  - **(O-2) Re-enable `mouse on`; teach Shift+drag for selection.** Restores wheel→tmux scrollback (the pre-regression behavior). Cost: left-drag selection now requires holding Shift (tmux captures plain drag). Reintroduces the exact UX Phase 60 removed.
  - **(O-3) Keep `mouse off`; intercept the wheel in xterm.js.** Add a `wheel` listener in `terminalInstance.ts`: when `alternate_on` (tmux) and tmux mouse is off, `preventDefault()` and send tmux copy-mode page keys instead of letting xterm alt-scroll. Most code, most winmux-specific, but keeps both native selection AND wheel scrollback. Needs live iteration.
- **DECISION (Yossi, 2026-06-22):** "גלגלת אמורה לגלול את הטקסט עם בחירה טבעית כמו שיש עכשיו; החיצים נשארים כמו עכשיו" → **O-3 (mouse off + wheel proxy).** Keep native selection; wheel scrolls; bare arrows stay = shell history.
- **DONE (2026-06-22): `90febca`.** Implemented O-3: xterm.js intercepts the wheel on the alternate buffer and sends **Alt+Up/Down** (CSI `1;3A`/`1;3B`); the bundled `winmux-tmux.conf` binds `M-Up`/`M-Down` (root + copy-mode + copy-mode-vi tables) to copy-mode scroll at a shell prompt (`#{alternate_on}==0`) and to plain arrows inside an alt-app. Gated on a per-instance `tmuxScrollProxy` flag (synced from the pane-persistence map) so non-tmux panes keep xterm's default; and on `g_winmuxTmuxWheelScroll` (= `use_winmux_tmux_config`). `remote-manifest.json` tmux-conf SHA refreshed (`3e85f8f5…`, 6150B) so the bootstrap re-uploads + verifies. **NEEDS YOSSI LIVE TEST:** reconnect a tmux pane (re-bootstraps the conf) → wheel-scroll at a prompt.   - remote target: `channel_open_direct_tcpip("localhost", remote_port, …)` — the REMOTE's own localhost (the service).
  - reverse control tunnel: `tcpip_forward("127.0.0.1", 0)`.
  - PortsWindow "Open" already used `http://127.0.0.1:<port>` (explicit, to dodge dual-stack).
- **Real bug found:** the in-app Browser (62.A C) built `http://localhost:<port>`. On dual-stack Windows `localhost` resolves to `::1` (IPv6) first, but the listener is IPv4-only — so it either fails to connect or, on fallback, the page ORIGIN (`localhost`) differs from the `127.0.0.1` origin PortsWindow uses → the service's CORS / cookie checks reject it. **Fix:** Browser now uses `http://127.0.0.1:<port>` to match PortsWindow.
- **Also:** `open_auto_forward` dlog now says "bound 127.0.0.1:<port> (loopback only …)" so a future external-IP report can be ruled out from the log immediately.
- **Deferred:** an opt-in "expose this port to the LAN" toggle (default off). Not built — the default IS loopback-only, which is what F wants; revisit only if Yossi wants to reach a port from a phone/tablet on his LAN. (If a remote service itself emits absolute URLs/redirects to its external IP, that's service-side config, outside winmux.)

### 2026-05-31 — Pipe listener pool (8 slots) + handler lifetime telemetry (Phase 44, `c65b0c5`)
- **Context:** Yossi reported sporadic `tunnel: bridge error … os error 231` on v0.2.5 — much less frequent than the pre-39.A storm, but still present. Diagnosis traced it to the rpc_server's accept loop: Phase 39.A raised `max_instances` to 254 and pre-created the "next" listener, but at any single moment there was still only ONE listener in accept state. Two clients racing for it within microseconds raced for that single slot; the loser got `ERROR_PIPE_NOT_AVAILABLE` (231). `max_instances(254)` is the OS upper bound on instances, NOT concurrent accept-ready listeners.
- **Decision:**
  - `LISTENER_POOL_SIZE = 8` simultaneous listeners, each in its own tokio task that loops `make_listener → connect() → tokio::spawn(handle_client_with_telemetry(...)) → recreate`. The handler runs on a separate task so the slot recreates its listener immediately — never blocked on handler duration. A burst of up to 8 concurrent clients serves instantly; the 9th+ still falls into tunnel.rs's bounded backoff (rare). max_instances(254) ceiling and the Phase 39.C `catch_unwind` are unchanged.
